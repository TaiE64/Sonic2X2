## 详录

### #1 肘部观测偏差 🔴(v1 作废)
- **症状**:无(隐性——LoRA 首层吸收常数偏差,训练照涨)。复查 obs 契约时发现。
- **根因**:`joint_pos` 观测项实为 `joint_pos_rel`(q−q_default 相对量),对齐 wrapper 按绝对量处理(减零位偏置)→ 肘部本体感受每帧偏 **+1.2 rad(69°)**。
- **修复**:①X2 默认姿态改为 Φ⁺(G1默认)(肘 −0.979/−0.985,肩roll ±0.139);②aligned_env 增加 pos_rel 规则。
- **教训**:相对量 vs 绝对量必须逐观测项核对 `terms/*.yaml` 的 func。

### #2 DOF 置换方向 + 兄弟字母序双重错 🔴(v1/v2 作废)
- **症状**:v2 启动打印 bias=1.19 异常。
- **根因**:①FrameMapper 查表方向用反(约定是 `q_mj=q_il[IL2MJ]`,按 `[il]=mj` 查了);②IsaacLab BFS **同层兄弟按字母序**排,按 URDF 文档序推导错了头/肩两层(6 槽);③自测用同一反向公式构造假环境 → **自洽错误互证**。
- **后果**:策略接的是 28/31 错位的机器人。
- **修复**:全部数组由真实 env 探针(probe_orders.npz)重生成;自测改用独立真值交叉验证。终验 bias 0.0097、31/31 对。
- **教训**:置换数组必须写明约定式,用**独立来源**真值验证;**永远别用同一公式构造测试环境**。

### #3 腕链语义错配 🔴(v3 作废)
- **症状**:动作缩放表暴露——按名配对的腕关节 scale 比值 5.65x / 0.16x。
- **根因**:G1 腕链 roll→pitch→yaw,X2 是 yaw→pitch→roll,按名字配对=前臂旋转接到末端小关节。
- **修复**:kinematic_alignment.py 加 SEMANTIC_JOINT_MAP(按链位配对);X2_ACTION_SCALE 覆盖为语义配对的 G1 逐关节值(29/29)。

### #4 std/熵死亡螺旋 🔴(v4 作废)
- **症状**:~10M steps 后 mean_reward 5→0.34 单调崩;entropy 14.7→15.7、std 0.38→0.42 单调升。
- **根因**:只有 LoRA+std 可训练时,熵奖励(coef 0.01)把"涨 std"当唯一得分捷径;任务梯度弱 → 熵项占优 → 噪声↑ → 奖励↓ 螺旋。**LoRA 适配特有坑**(论文全参训练任务梯度强压熵项)。
- **修复**:std 冻结在源值 0.385(论文本来就冻结全部预训练参数,冻结才是忠实)。
- **教训**:mean_reward 比分项奖励更早暴露崩塌;看曲线要配熵/std 一起看。

### #5 腕前臂轴符号翻转 🟠(v5b 作废)
- **症状**:FK 方向审计(双机器人逐关节 +δ 比子链世界轴)抓到 G1_wrist_roll↔X2_wrist_yaw **dot=−1.0**。
- **修复**:S_r 升级为带符号散射矩阵({−1,0,1});Φ⁺=S_rᵀ 因 sign²=1 仍精确。修后 29/29 对 dot≥0.985。
- **教训**:零位校准暴露不了符号翻转,必须做 FK 方向审计。

### #6 幻影 KL 🟠(v6–v9 优化信号被污染)
- **症状**:rollout 与 update 端同权重同 obs 的 logprob 恒差 ~0.19 → 假 KL → 自适应控制器把 LR 压死在 1e-5 地板。
- **根因**:多轮法医(padding 掩码、精度、权重逐位比、四路径二分)**始终未定位**。
- **修复**(工程免疫):ppo_trainer.py 在 epoch 循环前插 no_grad pre-pass,用 update 同路径重算全批 logprobs/old_mu/old_sigma 覆写 rollout_data → ratio 从精确 1、KL 从精确 0 出发。实测 |diff| 0.19→0.00000。
- **教训**:找不到根因时,数学上免疫掉它也是合法工程解。

### #7 漏了 D_r 髋解耦矩阵 🟡(论文 Level-2 不完整)
- **症状**:v10 终止原因脚部主导(ee_body_pos 0.38 / foot_pos_xyz 0.36)。
- **根因**:论文 Φ=J·D⁻¹·S(Eq.4/5),只实现了 S。G1 源髋 roll/yaw 轴绕 pitch 倾斜 **10.02°**,X2 髋正交。
- **修复**:build_hip_decoupling(D=R(+α),左右同号,FK 审计定向:R(−α) 膝方向 cos 0.998 最优);X2 neq=0 → J=I 本来就对。
- **效果**:零适配 SR 48.6→52.7%(+4.1pp),纯运动学修正。

### #8 训练终止判据比论文严 2–4 倍 🟠
- **根因**:sonic2x2 挂了 strict 训练组(anchor_ori 0.2/ee 0.15/anchor_pos 0.15 + 论文没有的 foot_pos_xyz 3D 0.2);论文 §4.1.3 = 0.25/0.8(≈46°)/0.3、无脚 3D。
- **修复**:新建 `terminations/tracking/paper_faithful.yaml`,v12 起训练也用它。
- **效果**:同 iter A/B——OMOMO SR 81.2→**94.8%**(+13.6pp)、LAFAN 66.4→70.8%。
- **教训**:hydra `+group=` 对已存 config 是加性合并删不掉旧项,要 `++` 直接覆盖参数。

### #9 LAFAN 评估没切窗 🟠(评估侧,曾造成"远不如论文"错觉)
- **根因**:LAFAN 是 57 条 4–5 分钟整条序列(中位 240s),OMOMO 才 ~5s;4 分钟零终止跟完不可能 → 假 1.8%。论文必然按短窗评。
- **修复**:切 300 帧(10s)窗,1266 窗全评(motion_lib_lafan_x2_win2)。
- **效果**:LAFAN SR 1.8%→**66.4%**(honest 全窗;只取首窗会虚高到 87.7%,不许用)。

### #10 GMR 上游:mink.solve_ik 限制传错位 🟡
- **根因**:第 6 位置参数是 `safety_break` 不是 `limits`,GMR 4 个调用点全传错 → IK 限位/速度/碰撞限制从未生效。
- **修复**:改 `limits=self.ik_limits`。普查 9380 条训练语料 0% 超限帧(转换管线有钳制),训练无恙。

### #11 LoRA critic 注入范围 ≠ S7 🟢
- v10–v12 注了整个 critic(backbone+in/out 投影),论文 Fig.7 S7 是 critic 仅 backbone。v13 改为 module.2/4/6/8/10。部署只用 actor → 对最终策略零影响,纯忠实性修正。
- 澄清:论文可训练占比 5.26% 是真实数字(Fig.6);我们 4.44%(v13)差异来自总参数分母不同(X2 39.1M vs Oli≈33M),非复现错。

### #12 root_rot 四元数格式混淆 🟢(诊断脚本)
- convert_x2_to_motion_lib 存的是 **xyzw**(注释写错),诊断脚本按 wxyz 喂 MuJoCo → 假"脚穿地/侧躺"警报。修复:`rr[[3,0,1,2]]`。
- 教训:四元数格式用重力方向判据验证,别信注释。

### #13 PLAY 块不加载 checkpoint 🟡(可视化)
- trainer(消费 +checkpoint 的)在 PLAY 路径上从不构造 → 之前所有 play 视频**全是零适配基线**。已在 PLAY 块内显式 load。
- 教训:渲染管线也要验证"权重真的进去了"。

### #14 🔴🔴 根因 BUG:body 映射表抄 G1、漏脚趾 → 参考骨架错乱(v1–v13 全部作废)

**这是"远远不如论文"的真正主因。用户从"关节点不对称"→"retarget错了"→"黄球对不上"→"骨架有问题"一路追问全对;我多次用不完整验证(只查腕/踝/头、按名索引绕过坏表)错误宣称"没问题"。**

- **症状**:训练参考 marker(黄点)上半身错乱:右肩 marker 落在左腕位置、左肩落在躯干、torso 落在 waist_yaw——**14 个追踪 body 里 10 个错**;正面看手臂 X 交叉。

- **为什么没早发现**:①DOF 表没中招(脚趾无自由度)→ 机器人能站能走;②5-point 主奖励错得温和(torso→waist_yaw、wrist_roll→wrist_yaw 距离都近)→ OMOMO 还能刷到 96%;③之前的"0mm FK 验证"用按名索引,恰好绕过了坏的数字表,且从没查肩。
- **修复(2026-07-10)**:从 probe_final.npz 真值重算两张表:
  `X2_MUJOCO_TO_ISAACLAB_BODY = [0,1,8,15,2,9,16,3,10,17,4,11,32,18,25,5,12,33,19,26,6,13,20,27,21,28,22,29,23,30,24,31]`(len 32);逆表 len 34,脚趾槽指父 ankle。

### 误报补录:"训练数据在平移/脚滑"(2026-07-10,提出当天被自己的插桩推翻)

- 用户看到机器人"老是脱离黄点",怀疑数据没处理好;我量出支撑期脚速 16cm/s(纯走路也如此),一度断言"GMR 根局部缩放导致结构性滑步,需 foot-lock 重做数据"。
- **三重证伪**:①GMR 源码根位移确实被 0.68 缩放(我的机制推导错误);②对比源 AMASS:源数据脚速(踝 12–33cm/s)比重定向后(8–17)更大——滑步是 AMASS 娘胎带的 mocap 噪声,GMR 反而衰减了它;③插桩分解:支撑段**净位移仅 ~1cm**(脚实际钉住),16cm/s 全是高频**抖动**,不驱动漂移。
- foot-lock 工具(footlock_cleanup.py)实测只降 1–2cm/s,留档搁置。**"脱离黄点"归因回到欠训练的世界系定位**(出生对齐 5cm→越走越漂→贴 0.25m 终止线生存)。
- 教训:看到异常均值,先分解"净位移 vs 抖动"再谈机制;"对比源数据"(用户的提议)是定位污染源的正确手段。

## sim2sim MuJoCo 部署重建(2026-07-11 凌晨,v13 wristfix @iter750)
